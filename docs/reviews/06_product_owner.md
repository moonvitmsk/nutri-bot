## 1. Анализ рисков ТЗ

### Критические риски (вероятность высокая / влияние высокое)

| # | Риск | Вероятность | Влияние | Примечание |
|---|------|-------------|---------|------------|
| R1 | Vercel timeout при Vision API (10-15 сек) | Высокая | Критическое | Serverless limit 10-30 сек, фото еды — ключевая фича |
| R2 | MAX API расхождение с документацией | Высокая | Критическое | Реальное поведение API неизвестно до интеграции |
| R3 | QR-коды ведут на Telegram, не MAX | Факт | Критическое | Бизнес-модель сломана прямо сейчас — блокер для монетизации |
| R4 | jsQR не декодирует фото с крышки бутылки | Средняя | Высокое | Низкое качество фото, угол съёмки, освещение — реальный кейс |
| R5 | OpenAI Vision: нераспознанная еда | Средняя | Среднее | Пользователь получит бесполезный ответ, отток |

### Значимые риски (средняя вероятность / высокое влияние)

| # | Риск | Вероятность | Влияние | Примечание |
|---|------|-------------|---------|------------|
| R6 | 4 агента в Этапе 2 — переусложнение архитектуры | Средняя | Высокое | Multi-agent = сложность отладки, нет пользы для MVP |
| R7 | Free trial 30 дней без конвертации в premium | Средняя | Высокое | Нет механики напоминания об окончании триала |
| R8 | Supabase cold start на Vercel | Низкая | Среднее | Может увеличить latency первого запроса |
| R9 | MAX: отсутствие sandbox / test environment | Высокая | Среднее | Тестирование только в проде |
| R10 | React/Vite админка — дублирует Supabase Studio | Низкая | Низкое | Overengineering на старте |

### Технический долг, заложенный в ТЗ

- Асинхронная обработка фото не описана (polling? webhooks?) — пробел в архитектуре
- Нет описания retry-логики при сбоях OpenAI
- Нет описания rate limiting для бесплатных пользователей на уровне кода
- Хранение лаб. результатов (Этап 2) — не указан формат и GDPR-совместимость

## 2. Пересмотр приоритетов MVP

### Что ДОЛЖНО войти в MVP (без этого продукт не работает)

1. **Webhook + MAX API интеграция** — основа всего
2. **Онбординг** — сбор данных пользователя (имя, цель, возраст)
3. **AI-чат с нутрициологом** — core value proposition
4. **Фото еды + Vision API** — дифференцирующая фича, но с учётом R1
5. **БД пользователей** — users, messages, food_logs в Supabase
6. **QR-активация подписки** — критично, потому что текущие QR сломаны

### Что НУЖНО убрать из MVP Этапа 1

- **Клавиатура (inline buttons)** — nice-to-have, текстовые команды достаточны для MVP
- **Детальная статистика** — Этап 2 правильно, но нужна хотя бы базовая (кол-во сообщений)

### Что нужно добавить в MVP (отсутствует в ТЗ)

- **Async обработка фото** — ответ "анализирую..." + отдельный webhook для результата (обход Vercel timeout)
- **Error handling + fallback сообщения** — что показывать при сбое OpenAI
- **Rate limiting** — не дать бесплатным пользователям исчерпать API бюджет
- **Конец триала — напоминание** — уведомление за 3 дня до конца 30 дней

## 3. Оценка scope

### Что избыточно для v1

| Элемент | Почему избыточно | Рекомендация |
|---------|-----------------|--------------|
| 4 агента в глубокой консультации (Этап 2) | Один качественный промпт заменяет 4 агента для 90% кейсов | Заменить на structured prompt с цепочкой |
| React/Vite Admin Panel (Этап 4) | Supabase Studio + Retool/Metabase покрывает 80% нужд | Postpone до v2, заменить Supabase Studio |
| QR-генерация в боте | QR генерируются на производстве, не в боте | Только чтение/валидация QR, не генерация |
| Логи в adminке (Этап 4) | Vercel Logs + Supabase Logs достаточно | Не строить кастомный логгер |

### Что недостаёт в ТЗ

| Элемент | Критичность | Обоснование |
|---------|------------|-------------|
| Async фото-обработка (очередь) | Критично | Vercel timeout — реальный блокер |
| Напоминание об окончании trial | Высокая | Основной conversion point |
| Механика graceful degradation при сбое OpenAI | Высокая | Продукт падает без fallback |
| Описание schema Supabase | Средняя | Без этого разные разработчики сделают несовместимо |
| Privacy policy / согласие на обработку данных | Высокая | Лаб. результаты — медицинские данные, юридический риск |
| Тест QR-декодирования с реальными крышками | Высокая | Качество фото в реальных условиях — неизвестная переменная |

## 5. User Stories для ключевых сценариев

### US-1: Первый запуск бота

```
Как новый пользователь Moonvit,
Я хочу запустить бота в MAX и пройти онбординг,
Чтобы получить персонализированные рекомендации по питанию.

Критерии приёмки:
- [ ] Команда /start запускает онбординг
- [ ] Бот последовательно запрашивает: имя, возраст, пол, цель (похудение/набор/здоровье), аллергии/ограничения
- [ ] Данные сохраняются в Supabase (таблица users)
- [ ] После онбординга пользователь получает персональное приветствие
- [ ] Если онбординг прерван — при следующем /start продолжается с места остановки
- [ ] Онбординг завершается за не более 5 шагов
```

### US-2: Анализ фото еды

```
Как пользователь с активной подпиской,
Я хочу отправить фото своего блюда,
Чтобы получить оценку калорийности и нутриентов.

Критерии приёмки:
- [ ] Бот принимает фото из чата MAX
- [ ] Немедленно отвечает "Анализирую ваше блюдо..." (не более 2 сек)
- [ ] Через 15-30 сек приходит ответ с: название блюда, КБЖУ, оценка качества питания
- [ ] Если фото нераспознано — бот предлагает уточнить текстом
- [ ] Результат сохраняется в food_logs с timestamp
- [ ] Free пользователи: не более 3 фото/день
```

### US-3: Активация premium через QR под крышкой

```
Как пользователь с продуктом Moonvit,
Я хочу отсканировать QR-код под крышкой в боте,
Чтобы активировать premium подписку.

Критерии приёмки:
- [ ] Пользователь отправляет фото QR-кода
- [ ] jsQR декодирует QR за не более 3 сек
- [ ] Система валидирует код в БД (одноразовый, не использован)
- [ ] При успехе: статус пользователя меняется на premium, приходит подтверждение
- [ ] При повторном использовании: сообщение "Этот код уже использован"
- [ ] При нечитаемом QR: "Не могу распознать код, попробуйте при лучшем освещении"
- [ ] Активация логируется с user_id, timestamp, qr_code_id
```

### US-4: Консультация нутрициолога

```
Как пользователь в free trial,
Я хочу задать вопрос нутрициологу на русском языке,
Чтобы получить персональный совет с учётом моего профиля.

Критерии приёмки:
- [ ] Любое текстовое сообщение (не команда) = вопрос нутрициологу
- [ ] Ответ учитывает данные онбординга (цель, ограничения)
- [ ] Ответ приходит за не более 10 сек
- [ ] Бот не даёт медицинских диагнозов (disclaimer в системном промпте)
- [ ] История последних 10 сообщений передаётся в контекст
- [ ] Лимит free: 10 вопросов/день, при исчерпании — предложение premium
```

### US-5: Конец trial — конверсия в premium

```
Как пользователь на 27-й день free trial,
Я хочу получить уведомление об окончании бесплатного периода,
Чтобы решить, активировать ли premium через QR-код.

Критерии приёмки:
- [ ] За 3 дня до конца trial приходит напоминание с объяснением преимуществ premium
- [ ] В день окончания trial — финальное сообщение с инструкцией по активации QR
- [ ] После окончания trial функции ограничены до 3 сообщений/день
- [ ] Бот не блокируется полностью — пользователь может задать 1 вопрос в день
- [ ] Напоминания отправляются не более 2 раз (за 3 дня и в день Х)
```

## 6. Критические зависимости и блокеры

### Граф зависимостей

```
[MAX API] ──────────────────► [Webhook] ──► [Онбординг] ──► [AI-чат]
     │                                                           │
     │                                              [Vision API async]
     │                                                           │
[QR на крышках] ──► [jsQR декодирование] ──► [Активация premium]
     │
[Переадресация Telegram→MAX] ◄── КРИТИЧЕСКИЙ БЛОКЕР (сейчас)
```

### Блокеры по категориям

**Внешние зависимости (вне контроля команды)**

| Блокер | Риск | Митигация |
|--------|------|-----------|
| MAX API нестабильность / расхождение с документацией | Критический | Spike 2-3 дня на изучение реального API перед планированием |
| OpenAI Vision API latency > 10 сек | Критический | Async architecture обязательна в MVP |
| QR-коды ведут на Telegram | Блокер монетизации | Немедленное действие: обновить landing QR или настроить redirect |

**Технические блокеры**

| Блокер | Риск | Митигация |
|--------|------|-----------|
| Vercel Serverless 10-30 сек timeout | Высокий | Либо async queue (Supabase Edge Function), либо VPS |
| jsQR: нечитаемые фото с крышек в реальных условиях | Высокий | Тест 20+ реальных фото ДО начала разработки |
| Supabase schema не описана в ТЗ | Средний | DB schema design как первый deliverable |

**Организационные блокеры**

| Блокер | Риск | Митигация |
|--------|------|-----------|
| Нет MAX developer account / sandbox | Высокий | Запросить доступ у VK немедленно |
| Лаб. результаты = медицинские данные | Юридический | Консультация юриста по 152-ФЗ до реализации Этапа 2 |

## 7. Definition of Done для MVP

### DoD — Функциональный уровень

- [ ] Webhook принимает сообщения от MAX и отвечает в течение 3 сек (acknowledgment)
- [ ] Онбординг проходит без ошибок для 95% пользователей (тест на 20 сценариях)
- [ ] AI-чат отвечает за < 10 сек в 90% случаев
- [ ] Фото еды обрабатывается асинхронно, пользователь получает результат
- [ ] QR-код с реальной крышки Moonvit активирует premium в 90% случаев при нормальном освещении
- [ ] Лимиты free tier работают и не дают превысить квоту

### DoD — Технический уровень

- [ ] Все переменные окружения в .env, не в коде
- [ ] TypeScript strict mode, нет any без обоснования
- [ ] Supabase schema задокументирована (migration files)
- [ ] Error handling: все API вызовы обёрнуты try/catch с fallback
- [ ] Логирование критических событий (активация QR, ошибки Vision)
- [ ] Vercel deployment работает без ручных правок

### DoD — Бизнес-уровень

- [ ] QR-коды на продукте ведут на MAX-бот (не Telegram)
- [ ] Пользователь проходит путь: первое сообщение → онбординг → вопрос → ответ за < 5 минут
- [ ] Активация premium через QR протестирована с реальными крышками (не mock)
- [ ] Продукт развёрнут на production URL (не localhost)

## 8. Roadmap на 6 месяцев

### Месяц 1: Foundation (Этап 1 + критические фиксы)

**Неделя 1 (Sprint 0 — Pre-dev)**
- Spike: MAX API реальное поведение (2 дня)
- Spike: jsQR с реальными крышками Moonvit (1 день)
- Решение по Vercel vs VPS для Vision API (1 день)
- Немедленный фикс: QR-коды → MAX бот (параллельно)

**Неделя 2-3 (Sprint 1 — Core)**
- Webhook + MAX API интеграция
- Онбординг flow
- Supabase schema (users, sessions, food_logs, qr_codes, subscriptions)
- AI-чат базовый

**Неделя 4 (Sprint 2 — Monetization)**
- Async обработка фото (Vision API)
- QR-декодирование и активация premium
- Free tier лимиты
- Error handling и fallback сообщения

**Milestone M1:** Рабочий бот с онбордингом, чатом, фото и QR-активацией

### Месяц 2: Enhancement (Этап 2)

- Анализ лаб. результатов (structured prompt, не multi-agent)
- Недельная статистика питания
- Напоминание об окончании trial
- Оптимизация промптов на основе реальных сессий

**Milestone M2:** Полный функциональный набор для premium пользователей

### Месяц 3: Stability & Analytics

- Мониторинг и алертинг (Vercel + внешний uptime)
- A/B тест промптов (контрольная / экспериментальная группа)
- Базовый дашборд (Retool или Supabase Studio настройка)
- Сбор и анализ первых данных: conversion rate free→premium

**Milestone M3:** Data-driven итерации, SLA 99%+

### Месяц 4: Scale & Retention

- Персонализированные рекомендации на основе истории
- Реферальная механика (поделиться с другом — дополнительные дни)
- Push-напоминания (если MAX API поддерживает)
- Расширенная статистика для пользователя (месяц, тренды)

**Milestone M4:** Retention механики запущены

### Месяц 5: Admin & Operations

- Admin panel v1 (если конверсия оправдывает, иначе Retool)
- Управление промптами без деплоя
- Управление QR-кодами (batch upload, статистика использования)
- Тарифные планы в коде (несколько уровней premium)

**Milestone M5:** Операционная независимость от разработчика

### Месяц 6: Expansion

- Анализ результатов 6 месяцев
- Глубокая консультация (multi-agent, если ROI оправдан)
- Интеграция с другими продуктами Moonvit
- Планирование v2 (новые мессенджеры, мобильное приложение?)

**Milestone M6:** Roadmap v2 готов на основе данных

## Итоговые рекомендации PO

### Топ-3 действия немедленно

1. **Обновить QR-коды** — перенаправить на MAX-бот или настроить redirect. Это блокирует монетизацию прямо сейчас.
2. **Провести Spike по MAX API** — 2 дня на реальное тестирование перед написанием кода. Расхождение документации и реальности = риск переделки.
3. **Решить вопрос async фото** — архитектурное решение (Supabase Edge Function queue vs VPS) нужно до начала Этапа 1, иначе придётся переписывать.

### Изменения в структуре этапов

- Этап 1: добавить async архитектуру и rate limiting как обязательные элементы
- Этап 2: заменить 4-агентную архитектуру на structured prompt chain (экономия 2-3 недель)
- Этап 3: QR-активация перенести в Этап 1 (это core монетизация, не опция)
- Этап 4: Admin panel — только после proof of business value в месяце 3-4

### Метрики успеха MVP (через 30 дней после запуска)

| Метрика | Цель |
|---------|------|
| Онбординг completion rate | > 60% |
| DAU (daily active users) | > 50 |
| Free → Premium conversion | > 15% |
| Среднее время ответа бота | < 8 сек |
| Uptime | > 99% |
| QR активаций / отгруженных крышек | > 5% |
