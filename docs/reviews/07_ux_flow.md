# UX Flow Review — NutriBot MAX v1.2

**Дата:** 2026-04-02
**Платформа:** Мессенджер MAX, inline-клавиатура
**Аудитория:** Российские пользователи, мобильные устройства

---

## 1. Карта пользовательских flow

### 1.1 Общая схема

```
/start
  └── Новый пользователь? ──YES──> [ОНБОРДИНГ 10 шагов]
  │                                        │
  │                                        └──> [ГЛАВНОЕ МЕНЮ]
  └──────────────NO──────────────────────> [ГЛАВНОЕ МЕНЮ]

ГЛАВНОЕ МЕНЮ
  ├── [Сфотографировать еду]   ──> context_state: awaiting_photo
  │       └── фото получено ──> AI-анализ ──> результат + кнопки действий
  │
  ├── [Что мне съесть?]        ──> AI-рекомендация на основе профиля
  │       └── результат ──> [Уточнить] / [Добавить в дневник] / [Меню]
  │
  ├── [Мой дневник за сегодня] ──> /today ──> сводка или empty state
  │
  ├── [Глубокий разбор]        ──> /deepcheck ──> 3-5 уточняющих вопросов
  │       └── финальный вопрос ──> AI-анализ ──> детальный отчёт
  │
  ├── [Загрузить анализы]      ──> context_state: awaiting_lab
  │       └── файл/фото ──> AI-разбор ──> рекомендации
  │
  └── [Мой профиль]            ──> /profile ──> сводка профиля + [Изменить]
```

### 1.2 Flow: Онбординг

```
/start
  │
  ├─ Шаг 1: Пол         [Мужской] [Женский]
  ├─ Шаг 2: Возраст     (свободный ввод или кнопки-диапазоны)
  ├─ Шаг 3: Рост        (ввод в см)
  ├─ Шаг 4: Вес         (ввод в кг)
  ├─ Шаг 5: Цель        [Похудеть] [Набрать массу] [Поддержать вес] [Здоровье]
  ├─ Шаг 6: Активность  [Сидячий] [Умеренный] [Активный] [Спортсмен]
  ├─ Шаг 7: Аллергии    [Орехи] [Лактоза] [Глютен] [Нет] + [Другое...]
  ├─ Шаг 8: Питание     [Всеядный] [Вегетарианец] [Веган] [Без свинины]
  └─ Шаг 9: Витамины    [Moonvit Energy] [Moonvit Slim] ... [Не принимаю]
                │
                └──> Профиль создан ──> Главное меню
```

### 1.3 Flow: Фото еды

```
[Сфотографировать еду]
  │
  ├── free-пользователь (0 фото/день)
  │       └──> [Недоступно] ──> [Попробовать Trial бесплатно]
  │
  ├── trial/premium
  │       └──> "Отправьте фото вашего блюда"
  │               └── фото получено
  │                       ├── AI успешно распознал
  │                       │       └──> карточка: название, КБЖУ, порция
  │                       │               ├── [Добавить в дневник]
  │                       │               ├── [Скорректировать порцию]
  │                       │               └── [Это другое блюдо]
  │                       │
  │                       └── AI не распознал
  │                               └──> ошибка + альтернативы (см. раздел 5)
  │
  └── лимит исчерпан (trial: 3/3, premium: 20/20)
          └──> уведомление + кнопка апгрейда
```

### 1.4 Flow: Глубокий разбор (deepcheck)

```
[Глубокий разбор] или /deepcheck
  │
  └──> Вопрос 1 из 3-5: "Как вы себя чувствуете последние дни?"
          └── ответ
                └──> Вопрос 2: "Есть ли жалобы: усталость, сон, кожа?"
                        └── ответ
                                └──> ... (до 5 вопросов)
                                        └──> AI-анализ (15-30 сек)
                                                └──> Детальный отчёт
                                                        ├── [Сохранить PDF]
                                                        ├── [Рекомендации]
                                                        └── [В меню]
```

### 1.5 Flow: Загрузка анализов

```
[Загрузить анализы] или /lab
  │
  ├── free/trial
  │       └──> [Недоступно] ──> [Перейти на Premium]
  │
  └── premium ──> context_state: awaiting_lab
          └──> "Отправьте фото или PDF анализов"
                  └── файл получен
                          └──> AI-разбор ──> маркеры + рекомендации
                                  ├── [Подробнее по маркеру]
                                  ├── [Добавить в профиль]
                                  └── [В меню]
```

### 1.6 Flow: QR-активация подписки

```
/subscribe или кнопка апгрейда
  │
  └──> context_state: awaiting_qr
          └──> "Отсканируйте QR с упаковки Moonvit"
                  └── QR-данные получены
                          ├── валидный QR ──> активация ──> приветствие Premium
                          └── невалидный ──> ошибка + [Ввести код вручную] / [Купить]
```

## 2. Онбординг: проблемы и предложения

### 2.1 Критические проблемы

**Проблема: 10 шагов без прогресс-индикатора**

Пользователь не знает, сколько осталось. По данным исследований (Duolingo, Headspace), онбординг с индикатором прогресса показывает completion rate на 23-31% выше.

Решение — добавить в каждое сообщение:
```
Шаг 3 из 9 — Рост

Укажите ваш рост:
```

**Проблема: Свободный ввод числовых данных (рост, вес, возраст)**

Пользователь может ввести невалидное значение: "175 см", "175.5", "сто семьдесят пять". Бот должен уметь обработать все эти варианты и дать чёткую ошибку при невалидном вводе.

Решение — кнопки-диапазоны для возраста, свободный ввод для роста/веса с валидацией и inline-подсказкой:
```
Введите рост в сантиметрах (например: 175)
```

**Проблема: Отсутствие возможности вернуться назад**

Ошибся на шаге 7? Нет кнопки [Назад]. Единственный выход — /reset, что обнуляет всё.

Решение — добавить [<< Назад] в каждый шаг онбординга.

**Проблема: Шаг с витаминами (шаг 9) — неочевидная цель**

Для нового пользователя неясно, зачем боту знать про витамины. Воспринимается как реклама Moonvit.

Решение — добавить контекст перед шагом:
```
Чтобы учесть нутриенты, которые вы уже получаете, укажите,
принимаете ли вы добавки Moonvit:
[Moonvit Energy] [Moonvit Slim] [Moonvit Basic]
[Другие добавки] [Не принимаю]
```

### 2.2 Средние проблемы

**Аллергии — сложный шаг без мультивыбора**

Если аллергий несколько — как выбрать? Inline-клавиатура MAX поддерживает до 210 кнопок. Нужен мультиселект с визуальной индикацией выбранного (например, галочка или [x] в тексте кнопки: `[✓ Орехи]`).

**Отсутствие skip для необязательных шагов**

Аллергии и витамины — не у всех. Нужна кнопка [Пропустить] или [Нет / Не знаю].

**Нет welcome-сообщения с ценностным предложением**

Первое, что видит пользователь после /start — сразу вопрос про пол. Нужно 1 экран:
```
Привет! Я NutriBot MAX — ваш персональный нутрициолог на основе ИИ.

За 2 минуты составлю ваш профиль и начну помогать с питанием.

[Начать] [Узнать подробнее]
```

### 2.3 Рекомендации по онбордингу

| Шаг | Текущее | Улучшение |
|-----|---------|-----------|
| Старт | Сразу вопрос | Welcome + value prop + [Начать] |
| Прогресс | Нет | "Шаг X из 9" в каждом сообщении |
| Назад | Нет | Кнопка [<< Назад] везде |
| Пол | Кнопки | Сохранить как есть |
| Возраст | Свободный ввод | Кнопки-диапазоны: до 25 / 26-35 / 36-45 / 46+ |
| Рост/Вес | Свободный ввод | Оставить + inline-подсказка формата |
| Аллергии | Одиночный выбор | Мультиселект + [Готово] |
| Витамины | Кнопки | Добавить пояснение зачем + [Не принимаю] |
| Финал | ? | Экран успеха с профилем-превью |

## 3. Навигация: оценка главного меню

### 3.1 Структура 3x2

```
[Сфотографировать еду]    [Что мне съесть?]
[Мой дневник за сегодня]   [Глубокий разбор]
[Загрузить анализы]        [Мой профиль]
```

**Сильные стороны:**
- 6 кнопок — оптимальное число для главного меню (Nielsen: 5-7 элементов)
- Глаголы действия ("Сфотографировать", "Загрузить") — понятные CTA
- Два столбца работают нормально на мобильном

**Слабые стороны:**

**Длина кнопок неравномерная** — "Сфотографировать еду" длиннее "Мой профиль" почти в 2 раза. На некоторых устройствах MAX кнопки с длинным текстом переносятся или обрезаются. Рекомендуется ограничить до 20 символов:

```
Текущее:              Предложение:
[Сфотографировать еду]  →  [Фото еды]
[Мой дневник за сегодня] →  [Дневник сегодня]
[Загрузить анализы]     →  [Загрузить анализы] (ок)
[Глубокий разбор]       →  [Глубокий разбор]   (ок)
[Что мне съесть?]       →  [Что мне съесть?]   (ок)
[Мой профиль]           →  [Мой профиль]       (ок)
```

**"Глубокий разбор" — неочевидное название для нового пользователя.** Не ясно, что это. Альтернативы: [Разбор самочувствия] или [Анализ состояния].

**Иерархия важности не отражена расположением.** По логике самая частая функция — фото еды, а самая редкая — профиль. Текущая сетка не приоритизирует. Предложение:

```
[Фото еды]         [Что съесть?]      <- частые действия (верх)
[Дневник сегодня]  [Глубокий разбор]  <- регулярные
[Загрузить анализы] [Мой профиль]     <- редкие (низ)
```

Порядок совпадает с текущим — это хорошо, просто текст кнопок нужно сократить.

**Отсутствует индикатор тарифа** — пользователь free нажимает [Сфотографировать еду] и получает отказ. Лучше сразу показывать в заголовке меню:

```
Привет, Александр! Тариф: Free (7/10 сообщений)

[Фото еды 🔒]      [Что съесть?]
[Дневник сегодня]  [Глубокий разбор]
[Загрузить анализы 🔒] [Мой профиль]
```

## 4. Состояния бота (context_state) — риски потери контекста

### 4.1 Текущие состояния

| Состояние | Триггер | Ожидаемый ввод |
|-----------|---------|----------------|
| `default` | main menu, /start | кнопки меню или команды |
| `awaiting_qr` | /subscribe | QR-данные или текст кода |
| `awaiting_lab` | /lab (premium) | фото или PDF файл |
| `awaiting_photo` | [Сфотографировать еду] | фото |

Отсутствуют: `awaiting_onboarding_step`, `awaiting_deepcheck_answer`, `awaiting_photo_correction`

### 4.2 Сценарии потери контекста

**Риск 1: Прерывание онбординга**

Пользователь начал онбординг на шаге 5 и закрыл приложение. Через день возвращается. Что происходит? Если сессия не сохранена — онбординг с шага 1. Если сохранена — непонятно, бот "помнит" или нет.

Решение — хранить `onboarding_step` в профиле пользователя. При возврате:
```
Вы не завершили настройку профиля. Продолжить с шага 5?

[Продолжить] [Начать заново]
```

**Риск 2: Текстовое сообщение в состоянии awaiting_photo**

Бот ждёт фото, но пользователь написал текст ("я съел борщ"). Большинство ботов либо игнорируют, либо выдают технический error.

Решение — graceful fallback:
```
Кажется, вы написали текст, а я ждал фото.

Хотите добавить блюдо вручную или всё же отправите фото?
[Добавить текстом] [Отправить фото] [Отмена]
```

**Риск 3: awaiting_lab — пользователь отправляет не тот файл**

PDF с договором вместо анализов. Или фото квитанции.

Решение — AI-проверка типа документа и сообщение:
```
Не похоже, что это медицинские анализы. Убедитесь, что
на фото видны результаты лабораторного исследования.

[Попробовать ещё раз] [Отмена]
```

**Риск 4: Команда /start в середине deepcheck**

Пользователь на шаге 3 из 5 глубокого разбора — вводит /start. Контекст разрушается.

Решение — интерсептор команд:
```
Вы проходите Глубокий разбор (шаг 3 из 5).
Если выйти сейчас — прогресс потеряется.

[Продолжить разбор] [Выйти в меню]
```

**Риск 5: Таймаут awaiting_qr / awaiting_lab**

Пользователь не отправил QR или файл. Через какое время сбрасывать состояние?

Рекомендация — TTL 30 минут с напоминанием через 10:
```
Вы хотели загрузить анализы. Файл ещё не получен.
[Загрузить сейчас] [Отмена]
```

## 5. Обработка ошибок — фото еды

### 5.1 Типология ошибок при распознавании фото

| Ситуация | Причина | Что показывать |
|----------|---------|----------------|
| Нечёткое фото | Размытость, тёмное освещение | Попросить переснять |
| Блюдо не распознано | Редкое/домашнее блюдо | Предложить ввести вручную |
| Несколько блюд на фото | Тарелка + гарнир | Предложить выбрать основное или разбить |
| Упаковка продукта | Этикетка вместо блюда | Переключиться на режим сканирования состава |
| Не еда | Предмет, лицо, пейзаж | Юмористичный/дружелюбный отказ |

### 5.2 Рекомендуемые сообщения об ошибках

**Нечёткое фото:**
```
Не удалось рассмотреть блюдо. Попробуйте:
- Сделать фото при хорошем освещении
- Держать телефон ровно над тарелкой
- Убедиться, что фото не размыто

[Переснять] [Ввести блюдо вручную]
```

**Блюдо не распознано:**
```
Hmm, не могу определить блюдо с уверенностью.

Это могло быть:
[Домашний суп] [Каша] [Другое блюдо]

Или напишите название блюда:
```

**Несколько блюд:**
```
На фото несколько блюд. Выберите, что записать:

[Суп] [Хлеб] [Всё вместе] [Ввести отдельно]
```

**Не еда:**
```
Это явно не обед! Отправьте фото блюда, и я посчитаю калории.

[Попробовать ещё раз] [В меню]
```

### 5.3 Принципы error UX

- Никогда не показывать технические ошибки ("Error 500", "AI service unavailable")
- Всегда давать альтернативный путь вперёд
- Сохранять тон: дружелюбный, не обвиняющий пользователя
- Ограничить число повторных попыток (3 попытки → предложить ввести вручную)

## 6. Empty states

### 6.1 /today — новый пользователь без записей

**Текущее (вероятно):** пустой список или "Нет данных"

**Проблема:** пустой список демотивирует. Новый пользователь думает, что бот не работает.

**Рекомендуемый empty state:**
```
Дневник за сегодня пуст

Начните отслеживать питание — это проще, чем кажется.

[Сфотографировать еду]  [Добавить вручную]

Совет: сфотографируйте завтрак — это займёт 5 секунд.
```

### 6.2 Empty states по сценариям

**После онбординга (первый вход в дневник):**
```
Ваш дневник питания пока пуст.

Норма калорий на сегодня: 1850 ккал
Осталось: 1850 ккал

Добавьте первый приём пищи:
[Сфотографировать] [Добавить вручную]
```

**/week — первая неделя:**
```
Статистика появится после 3 дней записей.

Пока у вас 0 дней из 3.

[Начать записывать] [Как это работает?]
```

**[Что мне съесть?] — профиль не заполнен:**
```
Чтобы дать рекомендацию, мне нужно знать ваши цели.

[Заполнить профиль] — займёт 2 минуты
```

**[Загрузить анализы] — free-пользователь:**
```
Разбор анализов доступен на тарифе Premium.

С Premium вы получаете:
- Расшифровку всех маркеров крови
- Персональные рекомендации по питанию и добавкам
- До 20 фото еды в день

[Попробовать Trial бесплатно] [Узнать о Premium]
```

## 7. Доступность (Accessibility)

### 7.1 Специфика мессенджера MAX

Мессенджер — не web-интерфейс, поэтому стандарты WCAG применяются ограниченно. Актуальные аспекты:

**Размер кнопок.** MAX рендерит кнопки согласно собственным гайдлайнам. Минимум touch target — 44x44pt по iOS HIG. Кнопки с текстом до 20 символов рендерятся комфортно.

**Контраст текста.** Форматирование HTML/Markdown: избегать светло-серого текста на белом фоне. Жирный (`**текст**`) для заголовков, курсив для второстепенного.

**Скринридеры.** В мессенджерах скринридер зачитывает текст сообщений целиком. Кнопки зачитываются как их текст. Значит:
- Кнопки типа `[>>]` или `[✓]` без текста — недоступны
- Эмодзи в кнопках зачитываются как "emoji" в некоторых скринридерах — избегать в кнопках, использовать в теле сообщений

**Голосовой ввод.** Российские пользователи часто используют голосовой ввод (Siri, Google Assistant, нативный MAX). Числовые поля (рост, вес) должны корректно обрабатывать "сто семьдесят пять" → 175.

**Длина сообщений.** Людям с когнитивными нарушениями трудно читать длинные блоки. Рекомендация — максимум 4-5 строк на одно сообщение, затем разбивка или кнопка [Подробнее].

### 7.2 Языковые аспекты

- Избегать канцелярита: "осуществить добавление" → "добавить"
- Числа: "1 850" с пробелом как разделителем тысяч (российский стандарт)
- Единицы: "ккал", "г", "мл" — без точки (стандарт Роскомстандарт)

## 8. Сравнение с best practices конверсионных чат-ботов

### 8.1 Референсы: Noom, MyFitnessPal, Cara Care, Welltory

| Критерий | Best Practice | NutriBot MAX (оценка) |
|----------|--------------|----------------------|
| Онбординг | Прогресс-бар + skip необязательных | Нет прогресса, нет skip |
| Первый экран | Value proposition + CTA | Сразу вопрос |
| Ошибки | Тёплый тон + альтернативный путь | Не определено |
| Empty state | Мотивирующий + action CTA | Вероятно пустой |
| Лимиты | Показывают остаток проактивно | Реактивно (при нажатии) |
| Уведомления | Push в нужное время (9:00, 13:00, 19:00) | Не в ТЗ |
| Прогресс | Еженедельные инсайты | /week (без email/push) |
| Персонализация | Обращение по имени, динамический контент | Базовый профиль |

### 8.2 Конверсионные паттерны, которых не хватает

**Streak механика (Duolingo-pattern).** "Вы записывали питание 5 дней подряд!" — простейший retention-триггер. Добавить в /today если streak > 2 дней.

**Social proof в upsell.** При показе paywall: "Более 12 000 пользователей Moonvit уже улучшили питание с Premium" — повышает конверсию на 18-24% по данным Noom.

**Loss aversion в уведомлениях.** "Вы не записали ужин — дневник за сегодня неполный" работает лучше, чем "Добавьте ужин".

**Anchoring в тарифах.** Показывать Premium первым, trial вторым — пользователь воспринимает trial как "бесплатную пробу дорогого", а не "ограниченную версию".

## 9. Топ-7 UX-улучшений

Приоритеты: **P0** — критично, влияет на retention; **P1** — высокий приоритет; **P2** — средний.

---

### #1 Прогресс-индикатор в онбординге [P0]

**Проблема:** пользователь не знает, сколько осталось шагов.
**Решение:** добавить "Шаг X из 9" в каждое сообщение онбординга + кнопку [<< Назад].
**Эффект:** ожидаемый прирост completion rate онбординга +20-30%.
**Сложность:** низкая.

---

### #2 Индикатор тарифа и лимитов в главном меню [P0]

**Проблема:** free-пользователь нажимает [Сфотографировать еду] и получает отказ — friction и frustration.
**Решение:** в заголовке меню показывать тариф и остаток. Заблокированные функции отмечать `🔒` прямо в кнопке.
**Эффект:** снижение frustration, потенциальный рост конверсии в trial.
**Сложность:** низкая-средняя.

---

### #3 Graceful fallback при нераспознанном фото [P0]

**Проблема:** если AI не смог распознать фото, пользователь в тупике.
**Решение:** 3 варианта действий (переснять / ввести вручную / выбрать из вариантов). Тёплый, не технический текст.
**Эффект:** удержание пользователей, которые иначе бросают бота.
**Сложность:** средняя.

---

### #4 Welcome-экран с value proposition [P1]

**Проблема:** новый пользователь сразу видит вопрос без контекста — зачем отвечать?
**Решение:** 1 экран до онбординга с описанием что умеет бот и кнопкой [Начать].
**Эффект:** снижение drop-off на первом шаге.
**Сложность:** низкая.

---

### #5 Мотивирующие empty states [P1]

**Проблема:** пустой дневник = ощущение что бот не работает.
**Решение:** для каждого пустого экрана — контекстный empty state с нормой дня, прогрессом и CTA-кнопкой.
**Эффект:** увеличение DAU и первого действия.
**Сложность:** низкая.

---

### #6 Сохранение состояния при прерывании сессии [P1]

**Проблема:** прерванный онбординг или deepcheck = потеря прогресса.
**Решение:** хранить `onboarding_step` и `deepcheck_answers` в профиле. Предлагать продолжить при возврате.
**Эффект:** снижение drop-off при прерывании.
**Сложность:** средняя.

---

### #7 Streak-механика и микро-уведомления [P2]

**Проблема:** нет retention-механики между сессиями.
**Решение:** показывать streak в /today ("5 дней подряд!"), send уведомления в 8:30 / 13:00 / 19:30 если не было записей.
**Эффект:** +15-25% к D7 retention (по данным аналогичных приложений).
**Сложность:** средняя-высокая (требует push или scheduled messages в MAX).
