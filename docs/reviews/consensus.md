# Мультиагентная экспертная оценка NutriBot MAX v1.2

**Дата:** 2026-04-02
**Объект:** Бот-нутрициолог NutriBot для мессенджера MAX (Moonvit)
**Источники:** ТЗ v1.2
**Методология:** 8 независимых экспертных оценок + консенсусное голосование

---

## 1. Клиент (UX онбординга)

### Что хорошо

- Онбординг линейный и понятный: имя - телефон - пол - возраст - рост - вес - цель - активность - аллергии - хронические - предпочтения
- Кнопки для пола, цели и активности -- не нужно вводить текстом
- Валидация числовых полей (возраст 10-120, рост 100-250, вес 30-300)
- Телефон можно пропустить словом "пропустить"
- После онбординга сразу показывается дневная норма КБЖУ и главное меню

### Проблемы

1. **11 шагов -- это много.** Пользователь открыл бота из любопытства, а его просят ввести рост, вес, аллергии, хронические заболевания. Высокий риск отвала на шагах 4-6. В ТЗ нет прогресс-бара ("шаг 3 из 10").
2. **Нет кнопки "назад".** Ввел неправильный возраст -- никак не вернуться. Приходится проходить заново через /start.
3. **Телефон на шаге 2 -- слишком рано.** Пользователь еще не понимает ценности бота, а его уже просят контактные данные. Это воспринимается как спам-сбор.
4. **Нет onboarding skip.** Нельзя пропустить весь онбординг и сразу задать вопрос боту. Фактически бот заблокирован до завершения всех шагов (router.ts строка 79: если `!user.onboarding_completed`, все сообщения уходят в `handleOnboardingStep`).
5. **Сообщение о возможностях бота** появляется только после завершения онбординга. Пользователь не знает, зачем ему это всё, пока не дойдет до конца.
6. **"Нет" для аллергий/хронических** -- неинтуитивно. Лучше кнопка "Нет аллергий" / "Пропустить".

### Оценка: 6/10

## 2. Маркетолог (стратегия привлечения)

### Текущая воронка

```
Покупка витаминов Moonvit
  --> QR под крышкой
    --> Фото QR в чат MAX
      --> Premium-доступ (безлимит)
```

### Что хорошо

- Модель "freemium через физический продукт" уникальна и создает петлю лояльности
- Trial 30 дней -- достаточный срок для формирования привычки
- Бесплатный функционал (чат + 3 фото/день) достаточен для первого впечатления

### Проблемы

1. **Разрыв между каналами.** QR-коды ведут на Telegram-бота (по ТЗ). Пользователь сканирует камерой -- попадает в Telegram, не в MAX. Инструкция "сфотографировать QR и отправить как фото в чат" неочевидна и требует отдельного обучения.
2. **Нет реферальной программы.** Пользователь не может пригласить друга. Нет механизма виральности внутри MAX.
3. **Нет onboarding-шага с объяснением QR.** Пользователь узнает о QR только через /subscribe или кнопку "QR-код". Нужно рассказывать про это в приветственном сообщении.
4. **Нет push-уведомлений / напоминаний.** Бот не напоминает о себе. Пользователь забудет через 3 дня.
5. **Лимит сообщений на free-тарифе** (`freeMessagesPerDay` в config) -- хорошо для конверсии, но текст "Достигнут лимит сообщений на сегодня" -- грубый. Нужен soft-wall с предложением ценности.
6. **Нет A/B тестирования.** Нет механизма для тестирования разных вариантов онбординга или промптов.

### Оценка: 5/10

## 3. UX Designer (flow бота)

### Анализ flow

```
/start
  |
  +--> [Онбординг: 11 шагов]
  |       |
  |       +--> Финиш: КБЖУ + главное меню
  |
  +--> [Главное меню: 6 кнопок в 3 ряда]
         |
         +--> Фото еды --> Vision --> подтверждение --> дневник
         +--> Deepcheck --> 4 AI-агента --> отчет
         +--> Анализы --> фото --> разбор
         +--> Дневник --> /today
         +--> QR-код --> фото --> активация
         +--> Профиль --> /profile
```

### Что хорошо

- Главное меню компактное: 6 кнопок в 3 ряда по 2 -- хороший баланс
- Flow фото еды логичен: отправил фото - получил анализ - подтвердил/отменил
- context_state грамотно переключает обработку фото (QR / lab / food)

### Проблемы

1. **Кнопка "Deepcheck" непонятна.** Обычный пользователь не знает, что это. Нет тултипа, пояснения. Лучше: "Полная консультация" или "Разбор питания".
2. **Длина сообщений.** `truncate()` обрезает до `config.limits.messageMaxLength`, но нет разбиения длинных ответов на несколько сообщений. AI-ответы часто > 4000 символов (лимит MAX API). Результат -- обрезка с "..."
3. **Нет "typing indicator".** Deepcheck занимает 20-30 секунд. Пользователь видит "Подожди 20-30 секунд" и ждет без обратной связи.
4. **Кнопка "Дневник" ведет на /today.** Но /week доступен только через текстовую команду. Нет кнопки для недельной статистики.
5. **После подтверждения еды** ("Записано в дневник!") нет предложения записать следующий прием пищи или перейти к дневнику.
6. **Ошибка в сообщении об ошибке.** chat.ts: "Произошла ошибка. Попробуй ещё раз." -- нет конкретики. Пользователь не знает, что пошло не так.
7. **Отсутствует /cancel.** Если пользователь начал отправлять анализы (context_state = awaiting_lab), он не может выйти из этого режима кроме как отправив фото.

### Оценка: 6/10

## 4. Data Analyst (метрики и дашборд)

### Текущее состояние

Из кода видно, что отслеживаются:
- `messages_today` (счетчик сообщений в день)
- `last_active_at` (обновляется при каждом сообщении)
- Food logs с `logged_at`
- Подписка `subscription_tier` + `trial_expires_at`

### Какие метрики необходимы

**Воронка онбординга:**
- Конверсия по каждому шагу (шаг 1 -> шаг 2 -> ... -> завершение)
- Время прохождения онбординга (от /start до финиша)
- Шаг отвала (на каком шаге бросают)
- % пользователей, пропустивших телефон

**Активность:**
- DAU / WAU / MAU
- Среднее количество сообщений в день на пользователя
- Среднее количество фото еды в день
- Retention: D1, D7, D30
- Cohort analysis по дате регистрации

**Конверсия:**
- Trial -> Premium (активация QR)
- Количество QR-активаций в день/неделю
- Среднее время от регистрации до QR-активации
- Повторные QR-активации (лояльность -- покупает второй продукт)

**AI-качество:**
- Среднее время ответа AI
- Токены на сообщение (есть в saveMessage)
- % ошибок AI (catch в chat.ts)
- % подтвержденных vs. отмененных анализов еды

**Что отсутствует в коде:**
1. Нет таблицы событий / аналитики. Все метрики нужно собирать из nutri_users и nutri_food_logs. Необходима отдельная таблица `nutri_events` (event_type, user_id, payload, created_at).
2. Нет трекинга шага отвала в онбординге. `onboarding_step` хранится, но без timestamp не видно, когда пользователь остановился.
3. Нет трекинга ошибок AI. catch в chat.ts логирует в console.error, но не сохраняет в БД.
4. Нет трекинга QR-ошибок (фото не содержит QR, QR уже использован и т.д.).

### Рекомендуемые таблицы для дашборда

```sql
CREATE TABLE nutri_events (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID REFERENCES nutri_users(id),
  event_type TEXT NOT NULL,  -- onboarding_start, onboarding_step_N, onboarding_complete,
                             -- message_sent, photo_sent, qr_scanned, qr_error,
                             -- ai_error, deepcheck_start, deepcheck_complete
  payload JSONB DEFAULT '{}',
  created_at TIMESTAMPTZ DEFAULT now()
);
```

### Оценка: 4/10

**Should Have (неделя 3-4):**

7. Недельный и месячный отчет по нажатию кнопки (не только через /week)

9. Перемещение шага "телефон" в конец онбординга или в профиль

**Could Have (месяц 2):**

**Won't Have (пока):**

## 6. Юрист (дисклеймеры и GDPR)

### Текущие дисклеймеры в коде

formatter.ts содержит 3 дисклеймера:

| Функция | Текст | Где используется |
|---------|-------|------------------|
| `disclaimer()` | "Бот не заменяет консультацию врача или диетолога." | Онбординг (welcome), /help |
| `labDisclaimer()` | "Интерпретация носит информационный характер. При серьезных отклонениях обратитесь к врачу." | Анализы |
| `vitaminDisclaimer()` | "Перед приемом БАД проконсультируйтесь со специалистом." | Рекомендации витаминов |

### Что хорошо

- /deletedata реализован -- пользователь может удалить данные
- Дисклеймер показывается при первом контакте
- Отдельный дисклеймер для лабораторных анализов

### Проблемы

1. **Дисклеймер при онбординге слишком мелкий** -- добавляется через `disclaimer()` к приветственному сообщению курсивом. Легко пропустить. Рекомендация: вынести в отдельное сообщение с кнопкой "Я понимаю".
2. **Нет согласия на обработку ПДн.** По 152-ФЗ (а не GDPR -- бот работает в РФ) необходимо получить согласие до сбора имени, возраста, веса, аллергий, хронических заболеваний. Это медицинские / биометрические данные.
3. **Нет политики конфиденциальности.** Должна быть ссылка на документ, описывающий: какие данные собираются, как хранятся, кому передаются (OpenAI API!), срок хранения.
4. **/deletedata удаляет сразу без подтверждения.** В commands.ts строка 122: сначала отправляется сообщение "Все данные удалены", затем вызывается `deleteUserData`. Нет подтверждения ("Ты уверен? Это действие необратимо"). Нет grace period.
5. **Данные передаются в OpenAI.** Профиль пользователя (возраст, вес, аллергии, хронические заболевания) отправляется в systemPrompt для AI. Пользователь должен быть уведомлен о трансграничной передаче данных.
6. **Нет возрастного ограничения.** Бот принимает возраст от 10 лет. Для несовершеннолетних (до 18 лет по РФ) требуется согласие родителя.
7. **Телефон сохраняется без шифрования.** phone хранится в JSONB профиля в открытом виде.
8. **Нет дисклеймера о том, что это не медицинское устройство.** Согласно 323-ФЗ, AI-система, анализирующая результаты лабораторных анализов, может подпадать под определение медицинского изделия.

### Оценка: 3/10 (высокие юридические риски)

### #1 — Исправить QR-механику ДО начала разработки

**Проблема:** QR-коды под крышками ведут на `t.me/moonvit_bot` — открывают Telegram, не MAX. Маркетолог оценивает потерю конверсии в 40-60%. Вся бизнес-модель не работает в текущем виде.

**Что делать:**
- Обновить QR-коды на URL вида `https://max.ru/...` (задача для производства — до любой разработки)
- Поднять кнопку [Ввести код вручную] как основной fallback в UX
- В онбординге явно объяснить: «Сфотографируй QR прямо здесь — не сканируй камерой»
- PO: перенести QR-активацию из Этапа 3 в Этап 1 (это монетизация, не фича)

---

### #2 — Юридический compliance: ПДн + реклама БАД

**Три блокера от Юриста (риски до 18 млн руб штрафов + уголовная ответственность):**

**2а. Supabase = нарушение ФЗ-152**
Supabase хранит данные на серверах в США/Ирландии. Данные о здоровье — спецкатегория (ст. 10 ФЗ-152). Штраф за трансграничную передачу без уведомления РКН: до 18 млн руб.
→ Решение: Yandex Cloud PostgreSQL или Supabase EU + уведомление РКН.

**2б. Функция /lab = риск незаконной медицинской деятельности (ст. 235 УК РФ)**
Текущий дисклеймер «информационный характер» — не акцепт и не защита.
→ Решение: обязательный явный акцепт перед `/lab`, переформулировать в «справка о референсных значениях».

**2в. Рекомендации SKU Moonvit = реклама без маркировки**
Требуются: ERID-токен (маркировка с 2023 г.) + фраза «БАД. Не является лекарственным средством» (ст. 25 ФЗ-38).
→ Добавить в `talking_points.md` каждого SKU, автоподставлять при рекомендации.

**Также:** добавить в онбординг «Шаг 0» — согласие на обработку ПДн с кнопкой «Принимаю».

### #3 — Редизайн онбординга: прогресс + бесплатность + быстрый старт

**Три конкретных фикса (упомянули 5 из 8 агентов):**

- **Прогресс-бар:** `Шаг 3 из 10 ●●●○○○○○○○` — completion rate +20-30% от одной строки (UX, Клиент)
- **Бесплатность в шаге 1:** «Первые 30 дней — бесплатно, без карты» — снимает тревогу в середине онбординга (Клиент, SEO, Маркетолог)
- **Quick start после КБЖУ:** вместо показа меню сразу предложить `[Что поесть на завтрак?]` `[Сфотографировать обед]` — момент пика вовлечённости нельзя терять (Клиент, UX)

### #4 — Async-обработка фото в MVP (не откладывать)

**Проблема:** OpenAI Vision API отвечает 10-15 сек. Vercel serverless: лимит 10-25 сек. Timeout при анализе фото еды и лаб. анализов — высокий риск в production.

**Что делать:**
- Паттерн: принять фото → «Анализирую...» → обработать асинхронно → отправить результат
- Либо Vercel Pro (`maxDuration: 60`) + streaming
- Fallback-сообщение при timeout: «Не успел — попробуй ещё раз»
- Data Analyst: алерт при error rate > 5% (SQL в `05_data_metrics.md`)

---

### #5 — Переработать тексты кнопок: от функций к выгодам

**Быстрая правка (1 час работы), высокое влияние на engagement (SEO, UX, Клиент, Маркетолог):**

| Текущий текст | Предложение |
|---------------|-------------|
| Сфотографировать еду | Что я сейчас ем? |
| Что мне съесть? | Что съесть сейчас? |

| Мой профиль | Мои цели и норма |

Добавить `🔒` на недоступные функции free-тарифа с подсказкой — пользователь не должен получать отказ после нажатия.
