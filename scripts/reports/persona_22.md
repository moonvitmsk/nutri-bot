## Bugs Found

### 1. [CRITICAL] edit_height и edit_age не сохраняются через /editprofile

**Воспроизведение:**
- `/editprofile` → `edit_height` → отправить "178"
- `/editprofile` → `edit_age` → отправить "10", "120", "5", "150"

**Ожидалось:** поля `height_cm` и `age` обновятся в БД.

**Фактически:** `height_cm = null`, `age = null` — после каждой попытки редактирования. Поле `weight_kg` обновляется корректно (72 → 300). Только `edit_height` и `edit_age` игнорируются молча, 200 OK возвращается, но данные не сохраняются.

**Дополнительно:** Первый вызов `/editprofile` → `edit_height` → "178" во время онбординга тоже не сработал, хотя height_cm = null при profile_skip ожидаемо.

### 2. [CRITICAL] context_state застрял в awaiting_food_text после /addfood в Day 7

**Воспроизведение:**
- `/addfood` → текст "финальный тест всех фич" → `confirm_food` → `/today` → `/week` → ... → `/deletedata` → `cancel_delete`

**Ожидалось:** после `cancel_delete` `context_state = idle`

**Фактически:** финальный `check_user` вернул `context_state = "awaiting_food_text"` — бот завис в состоянии ожидания еды. Одна из последовательностей команд нарушила машину состояний и не восстановила `idle`.

**Impact:** следующее сообщение пользователя будет интерпретировано как описание еды, а не команда.

### 3. [CRITICAL] SIM_GUIDE.md содержит неверное имя таблицы ошибок

**Воспроизведение:**
- `check_errors` (из sim-send.sh) обращается к `nutri_error_logs` (с `s` на конце)

**Фактически:** Supabase вернул:
```json
{"code":"PGRST205","hint":"Perhaps you meant the table 'public.nutri_error_log'"}
```

Реальная таблица — `nutri_error_log` (без `s`). SIM_GUIDE.md и sim-send.sh используют неверное имя — все вызовы `check_errors()` в симуляциях возвращают ошибку вместо данных.

### 4. [HIGH] Граничное значение height=250 молча отклоняется, но 100 принимается

**Воспроизведение:**
- `edit_height` → "100" → `check_user`: `height_cm = 100` (принято)
- `edit_height` → "250" → `check_user`: `height_cm = 100` (250 проигнорировано, осталось 100)

**Ожидалось:** либо 250 принять (реальный максимум), либо вернуть ошибку валидации пользователю.

**Фактически:** молчаливый отказ, HTTP 200, пользователь не получает фидбека. Верхняя граница не задокументирована.

### 5. [HIGH] Граничное значение weight=30 молча отклоняется, но 300 принимается

**Воспроизведение:**
- `edit_weight` → "30" → `check_user`: `weight_kg = 72` (30 проигнорировано)
- `edit_weight` → "300" → `check_user`: `weight_kg = 300` (принято!)

**Ожидалось:** 300 кг — физически нереальный вес, должна быть валидация с ошибкой.

**Фактически:** 300 принято и даже пересчитан `daily_calories = 6203` (с 2919 до 6203) — бот обрабатывает экстремальные значения без предупреждения. Нижняя граница 30 молча отклоняется.

### 7. [MEDIUM] Инъекции XSS и SQL принимаются без видимой санитизации (позитивный сигнал — но непрозрачно)

**Воспроизведение:**
- `/addfood` → `<script>alert('xss')</script>` → `confirm_food` → HTTP 200
- `/addfood` → `'; DROP TABLE nutri_food_logs; --` → `confirm_food` → HTTP 200

**Ожидалось:** либо rejection с пояснением, либо молчаливое принятие с санитизацией.

**Фактически:** оба запроса вернули 200 OK. Записи в `nutri_food_logs` не обнаружены через `check_food` — либо санитизация работает и данные не сохраняются, либо AI-анализ фильтрует нераспознанные продукты. **Поведение непрозрачно** — нельзя подтвердить факт санитизации без доступа к логам AI.

**Рекомендация:** проверить, что XSS-строки экранируются при хранении и при отображении в сообщениях бота.
